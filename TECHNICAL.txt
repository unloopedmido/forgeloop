ForgeLoop Technical Overview

ForgeLoop is an npm package named create-forgeloop. It is a maintenance-first scaffolder and command-line tool for discord.js Discord bots. The package publishes two executable entry points. The forgeloop binary runs the main CLI from the compiled dist/index.js file. The create-forgeloop binary runs dist/create.js, which wraps the same CLI core but normalizes argv so that bare arguments intended for scaffolding are treated as an implicit init subcommand.

Entry and routing

Both binaries ultimately call runCli in src/cli.ts with a string array of arguments. The create-forgeloop path first passes argv through normalizeCreateArgv in src/utils/create-entry.ts. If the user passes no arguments, the argv becomes a single element init. If the first token is already a known command name or is the help flag, argv is left unchanged. Otherwise init is prepended so that create-forgeloop my-bot behaves like forgeloop init my-bot.

runCli constructs an Output writer for stdout and stderr styling, then parses arguments. Short flags are expanded before parsing: -h maps to --help, -V to --version, -y to --yes, and -d takes the next token as --dir. The parser in src/utils/args.ts walks tokens: long flags support equals form or a following value; bare long flags become boolean true; unknown short flags that were not expanded are rejected.

The first positional token is the command. Subcommands and further positionals are preserved in ParsedArgs. Global flags --help and --version are handled before dispatch. Help without a command shows the top-level usage. Help with a command shows that command's spec from the registry. Core commands are dispatched from a COMMAND_HANDLERS map keyed by name. Unknown commands throw CliError with exit guidance.

Core commands are init, add, commands, remove, doctor, info, and docs. Their metadata and usage strings live in src/commands/registry.ts alongside CREATE_DIRECT_COMMANDS used by create-forgeloop normalization.

Project configuration and manifest

A ForgeLoop project is identified by a single config file at the project root named forgeloop.config.mjs. The constant CONFIG_FILE in src/constants.ts defines this name. resolveManifestLocation joins the chosen project directory with that filename and throws if missing. There is no fallback in code to other extensions or forgeloop.json; those appear only in older documentation or skill text.

The config file is loaded as an ES module via pathToFileURL and dynamic import. A cache-busting query parameter is set from the file mtime so repeated loads in one process see edits. The loader accepts the exported manifest object from default, config, or manifest named exports. The object is validated by validateManifest in src/manifest.ts, which enforces types, required fields, enums aligned with SUPPORTED_* constants in src/constants.ts, and consistency rules such as basic preset forbidding manifest.features.logging.

The in-memory shape is ForgeLoopManifest in src/types.ts, aliased as ForgeLoopConfig. The package also exports defineConfig from src/config.js as a typed identity helper for projects that import create-forgeloop/config.

createManifest in src/manifest.ts builds a manifest from InitOptions after init-time resolution. It sets paths.commandsDir, paths.eventsDir, and paths.interactionsDir to null for the basic preset and to conventional src paths for modular and advanced. paths.coreDir is set only for the advanced preset. Database features are null when database is none; otherwise prisma is required as ORM and the provider is sqlite or postgresql.

Init flow

runInit in src/commands/init.ts resolves options through resolveInitOptions in src/commands/init-options.ts. Non-interactive mode requires --yes and a project name. Interactive mode uses TTY prompts when available for name, language, preset, package manager, database and ORM pairing, tooling, and optional git, docker, ci, and install flags. Database none pairs with ORM none; any real database requires prisma. Defaults come from DEFAULTS in constants.

init calls buildProjectFiles in src/generators/project-files.ts with the manifest and options including the current CLI package name and a pinned cliPackageVersion of latest so generated package.json does not point at unreleased local versions. ensureDirectory and writeFiles materialize the tree. Optional git init and package manager install run afterward based on flags.

Project generation

buildProjectFiles composes many generators. package.json receives discord.js, dotenv, scripts for dev and start using tsx for TypeScript, and devDependencies including the ForgeLoop CLI package for TypeScript projects. Tooling files come from project-tooling.ts for eslint-prettier, biome, or none. Database wiring and Prisma files come from project-database.ts when enabled. Docker and GitHub Actions files come from project-ops.ts when flags are set. The root README is generated with setup steps. forgeloop.config.mjs is a JSON-stringified export default of the manifest.

Runtime source files are produced by src/generators/runtime.ts. Preset controls whether handler directories and interaction routing exist. Templates include the bot entrypoint, command and event loaders, optional core layout for advanced, interaction router and support libraries, logging helper when applicable, and initial sample handlers for modular and advanced shapes.

Preset semantics

basic yields a minimal bot without separate commands and events directories in the manifest; add, remove commands-related subcommands, and commands list or deploy are blocked at runtime with a clear error.

modular and advanced are handler projects: hasHandlerProject in src/utils/project.ts requires both paths.commandsDir and paths.eventsDir. advanced additionally sets coreDir for an extra structural area in templates.

Maintenance commands that need a project load the manifest with loadManifest or loadManifestWithLocation, resolve the project root via resolveProjectDir using --dir or cwd, and assert handler shape when generating or syncing application commands.

add and remove

runAdd dispatches on the first subcommand: command, context-menu, event, modal, button, or select-menu. It loads the manifest, asserts a handler project for generator subcommands, normalizes names, optionally prompts in TTY mode, validates Discord event names against generated metadata in src/discord/events.ts, and writes FileSpec entries through writeFiles. Slash and context menu modules live under manifest.paths.commandsDir. Events live under manifest.paths.eventsDir. Interaction handlers live under resolveInteractionsRoot, defaulting to src/interactions with subdirectories per kind.

Interaction templates support exact customId, RegExp with validated flags, or hand-written parseCustomId or matchCustomId modules; filenames derive from interactionFilePath and customIdToFileSlug in src/utils/interaction-paths.ts with length and charset validation.

runRemove deletes the corresponding files and can optionally re-run deploy for slash commands when --sync is passed, which reuses the same Discord credentials and target resolution as commands deploy.

commands subcommand

commands list loads every module file in the commands directory, imports each as an ES module, and collects commandModule.data.toJSON() into an array. Implementation detail: this runs inside the target project directory by spawning node with an eval script that reads the directory, imports each file, and writes JSON to file descriptor 3. stderr captures load errors. TypeScript projects require tsx in node_modules; the spawn arguments add --import tsx for ts. JavaScript projects use plain node. Before that, the CLI verifies discord.js is resolvable from the project.

commands deploy builds the same local payload, reads DISCORD_TOKEN and CLIENT_ID from process.env or a simple line-based parser of .env in the project root, and for guild scope GUILD_ID. assertDeployTargetFlags forbids both --global and --guild. resolveSyncTarget returns global if --global, guild if --guild, otherwise production NODE_ENV implies global and non-production implies guild. The Discord REST route is built in applicationCommandsRoute against discord.com/api/v10. Deployment uses HTTP PUT with the Bot token and replaces the entire command set for that scope.

commands diff fetches the current remote command array with GET, normalizes both sides for comparison by stripping read-only fields like id and version, sorts object keys, and compares per-command JSON shapes to classify local-only, remote-only, changed, and unchanged names.

putDiscordCommands and getDiscordCommands use fetch with AbortSignal.timeout capped at DISCORD_API_TIMEOUT_MS in src/lib/discord-app-commands.ts. Errors surface as CliError with status and parsed Discord error bodies when JSON.

doctor

runDoctor in src/doctor/run-doctor.ts loads manifest and package.json context, then runs an ordered list of check functions: config integrity, structure, environment, dependencies, discord workflow, optional network, and tooling. Default groups omit network and tooling unless --checks includes them. Issues carry severity error, warning, or info. --strict fails on warnings. --json prints a machine-readable report. --fix can regenerate missing scaffold-managed files from the manifest via buildProjectFiles, merge package.json scripts and dependencies, repair simple config export problems, and seed .env from .env.example when appropriate. Individual check implementations live under src/doctor/checks/.

info and docs

info prints a human-readable summary of the loaded manifest. docs opens DOCS_SITE_URL from constants in a platform-specific way using detached child processes such as open on macOS, xdg-open on Linux, wslview or Windows start under WSL, and similar fallbacks.

Errors and output

CliError carries an exitCode used to set process.exitCode in the catch path of runCli. Non-CliError Errors still print the message and exit with code 1. Output in src/utils/format.js centralizes banners, sections, colors, and plain lines for consistent CLI UX.

Build and publish

The TypeScript source compiles to dist via tsconfig.build.json. prepublishOnly runs typecheck, lint, test, and build. The published package includes bin, dist, README, and LICENSE per package.json files array. Node engine is at least 18.

Testing

Vitest drives unit and integration tests under tests/. Harness utilities simulate CLI runs and temporary project directories. Tests cover routing, init flows, doctor behavior, add and remove flows, and command-related integration.

Relationship to generated bots

Generated projects depend on discord.js and load slash command modules at runtime similarly to the CLI collection step: dynamic import of files under src/commands with a CommandModule contract exporting data and execute, optional autocomplete. Interaction routing in generated code scans modal, button, and select-menu subfolders and matches customId via exact map or flexible strategies with optional interactionRoutePriority ordering. Startup may sync commands to Discord based on NODE_ENV and environment variables; the README and templates document that behavior. ForgeLoop itself only orchestrates files and REST sync when you run commands deploy or remove with sync; it does not run your bot.

This document describes the implementation in the ForgeLoop repository as of the version in package.json and reflects behavior enforced by the TypeScript sources. If published human docs disagree, the source of truth for runtime behavior is the code paths described above.
